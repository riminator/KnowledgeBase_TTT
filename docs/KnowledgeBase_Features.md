# KnowledgeBase Feature List

## Overview

KnowledgeBase is a document and meeting intelligence platform built around document ingestion, semantic retrieval, grounded chat, and operational meeting workflows. It combines a React frontend, a FastAPI backend, pgvector-based search, configurable LLM providers, and Time Task Tracker integration.

## Core features

### 1. Multi-format document ingestion

KnowledgeBase can ingest a wide range of file types through the upload workflow in [`frontend/src/components/Upload.jsx`](frontend/src/components/Upload.jsx) and the backend endpoint in [`backend/api.py`](backend/api.py:105).

**What it does**
- Accepts document uploads through the web UI and API
- Processes uploaded files into searchable chunks
- Stores indexed content for later search and chat use
- Supports optional force re-indexing for already uploaded files

**Why it matters**
This feature creates the searchable knowledge layer that powers the rest of the platform.

### 2. File extraction and chunking

The extraction pipeline in [`backend/kb/extractors.py`](backend/kb/extractors.py) parses supported files and breaks them into retrieval-friendly chunks before embedding.

**What it does**
- Detects file type automatically
- Extracts text from supported document formats
- Splits content into chunks for vector indexing
- Preserves useful metadata such as meeting-related organizer and attendee fields when available

**Why it matters**
Chunking and extraction make long documents retrievable and usable by the semantic search and chat workflows.

### 3. Semantic embedding generation

Embeddings are generated by the logic in [`backend/kb/embedder.py`](backend/kb/embedder.py) using the configured provider.

**What it does**
- Converts text chunks into vector embeddings
- Supports cloud embedding mode with Nomic
- Supports local embedding mode with Ollama
- Produces vectors used for similarity search in pgvector

**Why it matters**
Embeddings are the foundation of semantic retrieval, enabling the system to find relevant content by meaning rather than exact keyword match.

### 4. Vector-based semantic search

The search workflow is implemented in [`backend/kb/search.py`](backend/kb/search.py:136), exposed by [`POST /search`](backend/api.py:123), and surfaced in [`frontend/src/components/Search.jsx`](frontend/src/components/Search.jsx).

**What it does**
- Accepts natural-language queries
- Finds the most relevant chunks using pgvector cosine similarity
- Returns scores, source paths, chunk indexes, and snippets
- Supports filters for file type, filename/source, and result count
- Lets users expand from snippets into full chunk content in the UI

**Why it matters**
This gives users a fast way to explore the indexed knowledge base without requiring full answer generation.

### 5. Retrieval-augmented chat

The RAG chat pipeline is implemented in [`backend/kb/chat.py`](backend/kb/chat.py), served through [`POST /chat`](backend/api.py:147), and used in [`frontend/src/components/Chat.jsx`](frontend/src/components/Chat.jsx).

**What it does**
- Accepts conversational questions over indexed content
- Includes prior conversation history in the request flow
- Retrieves relevant source chunks before answer generation
- Builds grounded prompts using retrieved context
- Returns answers together with source references and relevance scores
- Displays cited source chunks in the chat UI

**Why it matters**
This feature turns stored documents into a grounded conversational assistant instead of a generic chatbot.

### 6. Source inventory and source deletion

Source management is available through [`GET /sources`](backend/api.py:134), [`DELETE /sources`](backend/api.py:139), and the UI in [`frontend/src/components/Sources.jsx`](frontend/src/components/Sources.jsx).

**What it does**
- Lists indexed files and their chunk counts
- Shows source path and file type information
- Supports deleting all indexed chunks for a selected source
- Lets users refresh the indexed source inventory in the UI

**Why it matters**
This gives users operational control over the knowledge base contents.

### 7. Meeting transcript ingestion

Meeting-specific ingestion is implemented in [`frontend/src/components/MeetingUpload.jsx`](frontend/src/components/MeetingUpload.jsx) and [`POST /ingest-meeting`](backend/api.py:163).

**What it does**
- Accepts meeting transcripts and notes in formats such as TXT, MD, VTT, SRT, PDF, and DOCX
- Allows optional meeting metadata input including project code, organizer, and attendees
- Ingests meeting content into the same searchable knowledge base as other documents
- Supports re-indexing when a meeting file has already been uploaded

**Why it matters**
This extends the platform from document intelligence into meeting intelligence.

### 8. Automated meeting summarization

Meeting summarization is triggered inside [`backend/api.py`](backend/api.py:186) by reusing the RAG pipeline from [`backend/kb/chat.py`](backend/kb/chat.py).

**What it does**
- Generates structured summaries from meeting transcript content
- Prompts for topics discussed, decisions, action items, attendees, and duration
- Returns the generated summary to the UI after processing
- Shows which indexed source chunks were used to produce the summary

**Why it matters**
This converts raw transcripts into usable outputs for teams and downstream systems.

### 9. Time Task Tracker integration

The push workflow is implemented in [`backend/kb/pusher.py`](backend/kb/pusher.py:81) and is invoked from [`backend/api.py`](backend/api.py:193).

**What it does**
- Pushes meeting summaries into the `time_entries` table
- Stores meeting-related records with project code, task type, duration, date, description, meeting title, billable flag, confidence, status, organizer, and attendees
- Returns the inserted TTT entry identifier when available
- Captures push failures without blocking overall meeting ingestion

**Why it matters**
This makes meeting outputs operational by connecting summarization directly to time-tracking workflows.

### 10. Time Task Tracker retrieval in chat

The Time Task Tracker query layer in [`backend/kb/ttt.py`](backend/kb/ttt.py) is used by the chat orchestration in [`backend/kb/chat.py`](backend/kb/chat.py:112).

**What it does**
- Pulls relevant time-entry records into the chat context when a question suggests TTT lookup
- Supports project-based, billable, date-range, and meeting-history style retrieval
- Lets the LLM answer using both indexed documents and historical TTT data
- Formats TTT results as structured context for grounded responses

**Why it matters**
This allows KnowledgeBase to answer not only from uploaded files but also from operational time-tracking records.

### 11. Configurable LLM provider layer

LLM provider selection is handled in [`backend/kb/llm.py`](backend/kb/llm.py) and configured through [`backend/kb/config.py`](backend/kb/config.py).

**What it does**
- Supports local Ollama chat models
- Supports OpenAI-compatible providers
- Supports watsonx.ai through IAM-backed chat requests
- Centralizes provider switching behind a common `chat()` interface

**Why it matters**
This gives the platform deployment flexibility while preserving the same user-facing workflows.

### 12. Cloud-oriented deployment model

The project supports deployment across hosted frontend, backend, and database infrastructure, with configuration defined through environment-driven components such as [`backend/kb/config.py`](backend/kb/config.py) and deployment files like [`render.yaml`](render.yaml).

**What it does**
- Supports hosted frontend and backend deployment
- Supports managed PostgreSQL with pgvector
- Supports cloud-hosted embedding and LLM services
- Keeps behavior configurable through environment variables rather than code changes

**Why it matters**
This makes the platform practical for real-world hosted use while preserving local development flexibility.
